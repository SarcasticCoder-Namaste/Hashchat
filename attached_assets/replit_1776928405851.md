# HashChat - Real-time Messaging Platform

## Overview

HashChat is a modern, real-time messaging application built with React, Express, and WebSocket technology. It provides instant messaging, group chats, file sharing, message reactions, and comprehensive conversation management. The application uses a PostgreSQL database through Drizzle ORM and implements authentication via Replit's OpenID Connect system.

## User Preferences

Preferred communication style: Simple, everyday language.

## System Architecture

### Frontend Architecture

**Framework**: React with TypeScript, built using Vite for optimal development experience and production builds.

**UI Component System**: Built on shadcn/ui components (Radix UI primitives) with a custom design system featuring:
- Dark/light theme support with system preference detection
- Tailwind CSS for styling with custom color tokens and spacing scale
- Three-column desktop layout (sidebar, main chat, info panel) collapsing to two columns on mobile
- Responsive breakpoint at 1024px for layout adaptation

**State Management**:
- TanStack Query (React Query) for server state management and caching
- Local React state for UI interactions
- WebSocket connection for real-time updates

**Routing**: Wouter for lightweight client-side routing

**Key Design Decisions**:
- Component-based architecture with clear separation between chat UI, dialogs, and utility components
- Custom hooks for authentication (`useAuth`), WebSocket connections (`useWebSocket`), and theme management (`useTheme`)
- TypeScript for type safety across the entire frontend
- Path aliases (`@/`, `@shared/`) for cleaner imports

### Backend Architecture

**Framework**: Express.js with TypeScript running on Node.js

**Real-time Communication**: WebSocket server using the `ws` library, integrated with the HTTP server. Maintains client connections mapped by user ID for targeted message delivery and broadcasts.

**Session Management**: 
- Express sessions with PostgreSQL storage via `connect-pg-simple`
- Session-based authentication through Replit Auth (OpenID Connect)
- Passport.js for authentication strategy handling

**API Design**:
- RESTful endpoints under `/api` for CRUD operations
- WebSocket messages for real-time events (new messages, typing indicators, reactions, user status)
- Separation of concerns with dedicated route handlers in `server/routes.ts`

**File Handling**: Multer middleware for file uploads with 50MB limit, stored in local `uploads/` directory

**Key Design Decisions**:
- Monolithic backend structure for simplicity
- In-memory WebSocket client tracking (note: does not persist across server restarts)
- Logging middleware for request/response monitoring
- Development mode includes Vite integration for hot module replacement

### Data Storage

**Database**: PostgreSQL (via Neon serverless driver)

**ORM**: Drizzle ORM for type-safe database operations

**Schema Design**:
- `users`: Core user profiles with status, profile images, and metadata
- `contacts`: User-to-user relationships with nickname, pinned, and blocked states
- `groups`: Group chat metadata (name, description, image)
- `group_members`: Many-to-many relationship with role-based permissions
- `conversations`: Umbrella table for both direct and group conversations
- `conversation_participants`: Links users to conversations
- `messages`: Chat messages with support for replies, edits, and different content types
- `message_status`: Tracks delivery and read status per recipient
- `reactions`: Emoji reactions on messages
- `attachments`: File metadata linked to messages
- `sessions`: Session storage for authentication (required by connect-pg-simple)

**Rationale**: Normalized schema supporting both 1-on-1 and group conversations through a unified conversation model. Separate status tracking allows for granular read receipts. The design supports future features like message threading (via `replyToId`) and rich content types.

### Authentication & Authorization

**Provider**: Replit Auth using OpenID Connect

**Flow**:
1. Session-based authentication with secure HTTP-only cookies
2. OAuth discovery with token refresh capability
3. User profile data synchronized to local database on login
4. Middleware (`isAuthenticated`) protects API routes

**Security Considerations**:
- Sessions stored server-side in PostgreSQL with 7-day TTL
- HTTPS-only cookies in production
- Token refresh handled automatically by the OIDC client

**Rationale**: Leverages Replit's infrastructure for seamless authentication without custom user/password management. Session-based approach works well with WebSocket authentication.

## External Dependencies

### Third-Party Services

**Replit Auth (OpenID Connect)**: Primary authentication provider
- Discovery URL: `https://replit.com/oidc` (configurable via `ISSUER_URL`)
- Handles user identity and session management

**Neon Database**: PostgreSQL hosting
- Serverless PostgreSQL with WebSocket support
- Connection via `DATABASE_URL` environment variable

### Key Libraries

**Frontend**:
- `@tanstack/react-query`: Server state management and caching
- `wouter`: Lightweight routing
- `@radix-ui/*`: Accessible component primitives
- `tailwindcss`: Utility-first CSS framework
- `date-fns`: Date formatting and manipulation
- `lucide-react`: Icon library

**Backend**:
- `express`: Web framework
- `ws`: WebSocket server implementation
- `drizzle-orm`: Type-safe ORM
- `passport`: Authentication middleware
- `openid-client`: OIDC/OAuth client
- `multer`: File upload handling
- `express-session`: Session management
- `connect-pg-simple`: PostgreSQL session store

**Development**:
- `vite`: Frontend build tool and dev server
- `esbuild`: Backend bundling for production
- `tsx`: TypeScript execution for development
- `drizzle-kit`: Database migration tool

### Environment Variables

Required configuration:
- `DATABASE_URL`: PostgreSQL connection string
- `SESSION_SECRET`: Secret for session encryption
- `REPL_ID`: Replit application identifier (for auth)
- `ISSUER_URL`: OIDC provider URL (defaults to Replit)
- `NODE_ENV`: Environment mode (development/production)
